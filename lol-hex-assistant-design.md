# LOL 海克斯乱斗助手设计文档

## 1. 项目定位

LOL 海克斯乱斗助手是一款局内轻量游戏助手。

它在玩家进入海克斯乱斗后静默运行，通过 Riot / LOL 本地只读接口识别当前玩家英雄和等级，通过显示器级截图和 OCR 识别当前弹出的海克斯符文，再根据英雄和海克斯符文实时查询 ApexLOL 平台上的联动信息，并通过非侵入式透明展示层显示在海克斯符文卡片正下方。

核心目标不是替玩家自动选择，而是帮助玩家快速理解：

- 当前英雄和海克斯是否存在特殊联动；
- 这个海克斯为什么强或为什么弱；
- 当前三个海克斯里哪个更适合；
- 哪些海克斯属于陷阱选择；
- 重随后新的海克斯是否值得拿。

助手不做自动点击、不注入游戏进程、不 Hook、不读取游戏内存、不替玩家完成选择。

### 1.1 低可疑面约束

默认运行路径必须保持低可疑面：

```text
不注入
不 Hook
不读内存
不自动点击
不替玩家选择
```

核心链路只依赖本地只读接口、显示器级截图、用户校准区域、OCR 和非侵入式展示层。

以下能力不得进入默认运行路径：

```text
进程列表判断
窗口标题判断
游戏目录扫描
注入 / Hook
读取游戏内存
模拟点击或自动选择
```

进程枚举、窗口枚举、窗口标题读取和游戏目录扫描只能作为用户显式打开的诊断功能，用于排查环境问题，不参与英雄识别、截图、OCR、展示层定位或选择阶段判断。

## 2. 核心体验

玩家启动助手后，助手在后台静默运行。

当 Riot / LOL 本地只读接口可用，并返回当前局内玩家数据后，助手自动识别当前英雄和等级。

当玩家进入海克斯选择阶段时，助手通过显示器级截图，根据玩家首次校准的区域检测海克斯展示 / 隐藏按钮是否存在。

当海克斯卡片展开时，助手通过 OCR 识别三张海克斯符文名称，并用透明、置顶、点击穿透的展示层在三张符文卡片正下方分别展示对应说明卡片。

当玩家重随某个海克斯时，助手识别到该位置的海克斯名称变化，并只刷新对应位置的说明卡片。

当玩家收起海克斯面板时，助手隐藏说明卡片，但不认为海克斯已经选择完成。

当玩家完成当前海克斯选择后，如果三张卡片已经不可见，并且底部海克斯展示 / 隐藏按钮消失，助手才把按钮消失作为该轮选择完成的候选依据。

当玩家存在多档海克斯待选时，助手支持连续多轮识别和展示。

---

## 3. 评级体系

助手使用固定五档评级：

```text
夯爆了 ＞ 顶级 ＞ 人上人 ＞ NPC ＞ 拉完了
```

评级含义如下：

| 评级 | 含义 |
| --- | --- |
| 夯爆了 | 强联动，优先级极高，看到基本需要认真考虑 |
| 顶级 | 非常适配，通常值得拿 |
| 人上人 | 有明显收益，但看玩法、出装或熟练度 |
| NPC | 普通收益，不亏不赚，不是关键选择 |
| 拉完了 | 机制冲突、收益低、陷阱选择 |

界面展示时直接使用这套文案，不使用 S / A / B / C / D 这类传统评级。

## 4. 数据来源与查询方式

助手不完整爬取 ApexLOL 源站数据，也不在本地维护完整源站数据库。

运行过程中，助手根据当前英雄和 OCR 识别到的海克斯符文，实时拼接 ApexLOL 平台查询链接，获取当前需要的信息，再在本地解析并渲染。

基本链路如下：

```text
当前英雄
  +
OCR 识别到的海克斯符文
  ↓
拼接 ApexLOL 查询链接
  ↓
请求对应页面
  ↓
本地解析需要的信息
  ↓
生成助手展示卡片
```

助手界面需要明确声明：

```text
数据来源：ApexLOL
```

同时提供跳转到完整来源页面的入口。

## 5. 本地缓存策略

虽然助手不完整爬站，但需要做运行缓存，以降低重复请求、减少等待时间，并提升局内体验稳定性。

缓存粒度为：

```text
英雄 + 海克斯符文
```

示例缓存结构：

```json
{
  "champion": "Vi",
  "augment": "吞噬灵魂",
  "rating": "顶级",
  "tags": ["护盾", "生命值", "稳定收益"],
  "summary": "与蔚的被动护盾和进场节奏适配，属于稳定收益。",
  "source": "ApexLOL",
  "sourceUrl": "...",
  "fetchedAt": "2026-04-25T00:00:00+09:00"
}
```

运行策略：

```text
命中缓存：
  立即展示缓存内容

未命中缓存：
  请求 ApexLOL
  解析后展示
  写入本地缓存

请求失败：
  展示暂无数据
  保留来源跳转入口
```

## 6. 英雄与等级识别

英雄和等级通过 Riot / LOL 本地 Live Client Data API 获取。

优先使用本地只读接口，例如：

```text
GET https://127.0.0.1:2999/liveclientdata/activeplayer
```

接口返回当前玩家数据后，助手读取当前英雄、等级和后续状态机需要的基础信息。

默认判断方式是“本地接口是否可用”，不是“进程是否存在”。助手不通过进程扫描、窗口标题或游戏目录扫描判断玩家是否已经进入 LOL。

用途包括：

- 识别当前玩家英雄；
- 判断当前玩家等级；
- 根据等级生成待处理海克斯档位；
- 作为 ApexLOL 查询条件之一；
- 作为海克斯推荐说明的上下文。

海克斯弹窗本身不依赖 LOL API，因为已经确定 LOL 没有提供稳定的海克斯弹窗 API。

## 7. OCR 总体原则

OCR 不全局持续运行。

当前 OCR 落地目标为 **RapidOCR + ONNXRuntime**，通过 Python sidecar（`scripts/ocr_sidecar.py`）调用。离线回归评估（`scripts/eval-rapidocr.mjs`）在 5 张 2560×1440 真实截图上达到 15/15 exact match，未使用符文名错字表或样本特化纠错。模型完全本地离线，不依赖在线服务。

Tesseract 只保留为调试基线，在 `scripts/replay-ocr-debug.mjs` 的 Tesseract 分支中使用，不能靠针对单个样本的错字映射、符文名特化混淆字表或低置信度强行匹配来制造通过结果。

整体原则是：

```text
长期轻量监听
短时间截图探测
必要时才 OCR
```

普通状态下，助手只轮询本地只读接口状态、当前英雄和等级，不截图、不 OCR。

只有当玩家等级达到海克斯触发档位，并且存在未完成的海克斯选择时，助手才进入待领取状态。

待领取状态下，助手通过显示器级截图检测海克斯卡片和展示 / 隐藏按钮。卡片 / 名称区域可见优先于按钮区域，因为鼠标悬停符文时，详情浮层可能遮挡底部按钮区域。

只要三张海克斯卡片展开，助手就可以对三个名称区域进行 OCR；按钮区域被详情浮层遮挡不能阻止 OCR。

截图设计以显示器级截图为默认路径。窗口捕获不作为核心方案，这样可以避免默认链路依赖进程、窗口句柄或窗口标题。

截图策略如下：

```text
按用户选择的显示器截图
按校准区域裁剪按钮和卡片区域
先做黑屏检测
必要时延迟 100 - 300ms 后再次截图
对画面中心和校准区域做采样，判断截图是否有效
只有截图有效且按钮 / 卡片状态满足条件时才 OCR
```

### 7.1 当前全屏风险记录

以下只记录 2026-04-26 当前实测证据，不代表所有机器、地图或后续版本的永久结论。

当前诊断样本显示，莲华栈桥、嚎哭深渊、屠夫之桥三张地图的无边框模式都通过了连续截图刷新验证；独占全屏模式没有通过连续两次刷新标准。莲华栈桥全屏拿到过无边框旧帧，嚎哭深渊和屠夫之桥全屏首张非黑屏但第二张命中旧帧。

因此当前 MVP 的 OCR、Overlay 和状态机真实验收应优先以无边框模式为默认可用前提。独占全屏继续保留为风险和研究项；如果用户选择独占全屏，助手必须继续执行黑屏和旧帧判断，不能只按非黑屏结果进入后续 OCR 或 Overlay 流程。

## 8. 首次运行校准向导

由于不同玩家的分辨率、显示模式、UI 缩放、显示器比例和多显示器布局不同，助手首次运行时需要引导玩家完成区域校准。

校准流程如下：

```text
1. 玩家打开 LOL
2. 玩家进入一局海克斯乱斗
3. 等待海克斯选择界面出现
4. 玩家选择当前游戏所在显示器
5. 助手执行显示器级截图
6. 玩家在截图上框选关键区域
7. 助手执行 OCR 测试
8. 玩家确认后保存校准配置
```

校准完成后，助手后续根据这些区域进行 OCR、按钮检测和展示层定位。校准截图来自显示器级截图，不要求绑定具体游戏窗口。

---

## 9. 校准区域

首次校准需要框选三类区域：

```text
1. 海克斯名称 OCR 区域
2. 海克斯卡片底部锚点区域
3. 海克斯展示 / 隐藏按钮区域（折叠判定）
```

### 9.1 海克斯名称 OCR 区域

玩家需要分别框选左、中、右三张海克斯卡片的名称区域。

框选要求：

```text
宽度尽量接近整张海克斯卡片宽度
高度约为文字高度的 3 倍
上下保留一定空白
不要框到图标、描述正文、按钮或边框
```

这样可以避免某些海克斯名称较长时识别不全。

该区域用途：

```text
裁剪名称区域
执行 OCR
识别海克斯名称
判断重随后是否发生变化
```

### 9.2 海克斯卡片底部锚点区域

玩家需要分别框选左、中、右三张海克斯卡片的底部区域。

该区域用途：

```text
决定助手说明卡片显示在哪里
计算展示层 x / y / width
让说明层贴在对应海克斯卡片正下方
```

展示层位置计算方式：

```text
展示层 x = 底部锚点 x
展示层 y = 底部锚点 bottom + 间距
展示层 width = 底部锚点 width
```

### 9.3 海克斯展示 / 隐藏按钮区域

玩家需要框选屏幕底部中间的海克斯展示 / 隐藏按钮。

该区域用于在卡片不可见时辅助判断玩家是否仍处于海克斯选择阶段，不能作为面板展开状态的唯一依据。

核心规则：

```text
卡片 / 名称区域可见优先于按钮区域
卡片消失 ≠ 选择完成
卡片不可见时，按钮仍存在 = 面板收起或入口还在
卡片不可见时，按钮消失 = 当前海克斯选择阶段结束候选
```

因为玩家可以点击按钮收起海克斯面板。面板收起后，三张海克斯卡片不可见，但符文选择并没有完成。

鼠标悬停海克斯符文时，详情浮层可能遮挡底部中间按钮区域。此时如果三张卡片和名称区域仍可见，助手必须保持面板展开，允许 OCR 和展示层，不能因为按钮区域不可见而标记当前轮完成。

## 10. 坐标存储方式

校准区域不只存绝对坐标，还要存归一化坐标。

示例：

```json
{
  "xRatio": 0.428,
  "yRatio": 0.354,
  "widthRatio": 0.145,
  "heightRatio": 0.048
}
```

运行时根据当前截图尺寸还原真实坐标：

```text
realX = screenshotWidth * xRatio
realY = screenshotHeight * yRatio
realWidth = screenshotWidth * widthRatio
realHeight = screenshotHeight * heightRatio
```

这样可以提高不同分辨率下的兼容性。

如果玩家修改游戏分辨率、显示模式、UI 缩放或更换显示器，需要提供重新校准入口。

## 12. 海克斯档位与待处理队列

海克斯选择档位为：

```text
3 级 / 7 级 / 11 级 / 15 级
```

玩家可能长期存活，导致一次性积累多个未选择档位。

例如：

```text
当前等级：11
已完成：3
未完成：7、11
待处理队列：[7, 11]
```

因此助手不能只维护单个 pendingMilestone，而需要维护待处理队列：

```text
pendingMilestoneQueue
```

生成逻辑：

```ts
const MILESTONES = [3, 7, 11, 15];

function getPendingMilestones(level: number, completed: Set<number>) {
  return MILESTONES.filter(m => level >= m && !completed.has(m));
}
```

## 13. 海克斯选择阶段判断

当前确定规则如下：

```text
用等级判断是否有资格
用待处理队列判断是否还有未完成档位
用卡片判断当前面板是否展开
卡片不可见时，再用按钮判断是面板收起还是阶段结束候选
用名称变化判断是否发生重随
```

进一步拆分：

```text
等级达到 3 / 7 / 11 / 15：
  说明玩家可能有海克斯待领取

存在未完成档位：
  说明助手需要继续监听

卡片展开：
  说明面板展开，可以 OCR 三个海克斯名称；按钮区被详情浮层遮挡也不影响判断

卡片收起：
  说明暂时不 OCR，不展示说明层，但不能标记完成

卡片不可见且按钮存在：
  说明玩家仍处于海克斯选择阶段，面板可能被收起

卡片不可见且按钮消失：
  说明当前海克斯选择阶段可能结束
```

## 14. 状态机设计

推荐状态机如下：

```text
IN_GAME_MONITORING
  普通监听状态

AUGMENT_ELIGIBLE
  等级已满足，存在未完成海克斯档位

AUGMENT_STAGE_ACTIVE
  检测到海克斯展示 / 隐藏按钮，确认处于选择阶段

AUGMENT_COLLAPSED
  按钮存在，但三张海克斯卡片不可见

AUGMENT_EXPANDED
  三张海克斯卡片可见；按钮区域可能被详情浮层遮挡

AUGMENT_ROUND_COMPLETED
  当前这一轮选择完成，但可能还有下一档

AUGMENT_STAGE_COMPLETED
  所有待处理档位完成，按钮消失，退出选择阶段
```

## 15. OCR 触发策略

### 15.1 普通监听状态

在普通监听状态下：

```text
只轮询本地只读接口状态、当前英雄和当前等级
不截图
不 OCR
```

### 15.2 等级满足后

当玩家等级达到 3 / 7 / 11 / 15，且存在未完成档位时：

```text
进入 AUGMENT_ELIGIBLE
开始低频显示器级截图，探测海克斯卡片区域和展示 / 隐藏按钮区域
```

此时只做轻量状态探测，不 OCR 三张海克斯卡片。若已经确认卡片展开，再进入 OCR 流程。

### 15.3 卡片或按钮存在

三张卡片展开时，优先认为面板展开；鼠标悬停符文详情可能遮挡按钮区域，不能因此阻止 OCR。

```text
卡片展开：
  OCR 三个名称区域
  查询 ApexLOL
  展示说明层

卡片收起：
  隐藏说明层
  不标记完成
  持续等待重新展开
```

### 15.4 按钮消失

只有满足以下条件，才认为当前海克斯选择阶段结束：

```text
三张卡片不可见
本阶段曾经检测到按钮存在
后续按钮连续 N 次检测不到
待处理队列为空或当前轮已完成
```

按钮消失后，助手隐藏说明层，并根据待处理队列决定是否回到普通监听状态。

## 16. 多轮海克斯选择处理

当玩家一次性存在多个未完成档位时，助手需要支持连续多轮选择。

示例：

```text
当前等级：11
已完成：3
待处理队列：[7, 11]
```

处理流程：

```text
检测到按钮存在
进入海克斯选择阶段

第一轮：
  识别三张海克斯
  展示说明层
  玩家选择
  当前轮完成
  标记 7 级已完成

继续监听
发现还有 11 级未完成

第二轮：
  识别新三张海克斯
  展示说明层
  玩家选择
  标记 11 级已完成

待处理队列清空
按钮消失
退出海克斯选择阶段
```

关键规则：

```text
卡片消失 = 可能是面板收起，不能单独判断当前轮完成
卡片不可见后的按钮消失 = 整个海克斯选择阶段结束候选
```

---

## 17. 当前轮完成判断

一轮选择完成和整个阶段完成需要分开判断。

当前轮完成条件：

```text
本轮曾经成功识别到有效三张海克斯
之后三张卡片消失或刷新为下一组三张海克斯
当前存在待处理档位
```

当前轮完成后：

```text
标记当前 milestone 已完成
从 pendingMilestoneQueue 中移除该 milestone
如果队列还有后续档位，继续等待下一轮海克斯
如果队列为空，等待按钮消失后退出选择阶段
```

## 18. 面板收起处理

如果玩家点击底部按钮收起海克斯面板，会出现：

```text
按钮仍存在
三张卡片不可见
```

如果鼠标悬停符文详情遮挡按钮区域，但三张卡片仍展开，这不是面板收起，助手必须继续 OCR 和展示说明层。

此时助手行为：

```text
隐藏助手说明层
不标记当前轮完成
不标记当前阶段完成
持续监听按钮和卡片状态
```

当玩家再次展开面板：

```text
重新识别三张海克斯
重新展示说明层
```

## 19. 重随处理

每个海克斯符文可以重随一次。

在 `AUGMENT_EXPANDED` 状态下，助手持续低频识别三个名称区域。

推荐频率：

```text
每 800 - 1200ms OCR 一次
```

处理逻辑：

```text
读取当前三个名称区域
与上一次识别结果对比

如果某个 slot 的海克斯名称变化：
  只刷新该 slot 的展示内容

如果某个 slot 暂时识别失败：
  保留旧内容
  连续多次失败后再进入异常处理
```

示例：

```text
slot1：量子计算 → 不变
slot2：吞噬灵魂 → 巨像勇气
slot3：双刀流 → 不变
```

只更新第二张卡片下方的说明层。

## 20. 展示层设计

展示层目标是透明、置顶、点击穿透的非侵入式展示层，直接显示在对应海克斯符文卡片正下方。

展示层不得拦截鼠标，也不得影响玩家操作：

```text
不影响重随按钮点击
不影响收起 / 展开海克斯面板
不影响选择海克斯
不模拟点击
不替玩家选择
```

展示层只负责显示说明。玩家鼠标和键盘操作必须继续落到游戏本身。

布局：

```text
[海克斯卡片 A]   [海克斯卡片 B]   [海克斯卡片 C]
[助手说明 A]     [助手说明 B]     [助手说明 C]
```

每张说明卡片控制在短内容内。

推荐结构：

```text
夯爆了
蔚 × 量子计算
Q / R 位移触发收益极高
爆发流优先，操作不熟练可降档
数据来源：ApexLOL
```

另一个示例：

```text
拉完了
蔚 × 双刀流
机制收益低，容易和技能节奏冲突
不推荐优先选择
数据来源：ApexLOL
```

### 20.1 独占全屏 Overlay 风险

独占全屏下，透明置顶展示层可能无法稳定覆盖游戏画面，也可能被系统、驱动或游戏显示模式遮挡。

因此独占全屏展示层必须先做 POC，验证：

```text
是否能稳定显示在游戏画面上方
是否保持点击穿透
是否不影响重随、收起 / 展开面板、选择海克斯
是否在切屏、分辨率变化、窗口焦点变化后恢复正常
```

如果非侵入式方案不能稳定覆盖，不得转向注入、Hook 或读取内存。兜底路线只能回到：

```text
建议用户使用无边框显示模式
使用无边框覆盖层
使用独立助手窗口展示结果
使用手动输入模式
```

## 22. OCR 纠错

助手需要维护海克斯符文词库。

OCR 结果不直接使用，而是先进行保守的标准名称匹配。

流程：

```text
校准区域裁剪
  ↓
名称行定位和多种预处理候选
  ↓
OCR 引擎输出候选文本和置信度
  ↓
海克斯词库模糊匹配
  ↓
高置信度且相似度达标时得到标准海克斯名称
  ↓
用于查询 ApexLOL
```

低置信度时：

```text
显示疑似结果
允许用户手动修正
保留裁剪图、增强图、候选文本和匹配分
```

禁止把某张样本识别错的字直接写成全局纠错规则。允许使用通用归一化、编辑距离、包含匹配和图像预处理候选排序；不允许使用符文名特化混淆字表、单样本错字映射或低置信度强制通过。

后续必须以多张真实无边框样本验证 OCR 引擎，覆盖不同地图、画质、缩放和 UI 状态。如果 Tesseract 在这些样本中继续低稳，应切换到本地 OCR 引擎适配层，而不是继续堆样本补丁。

示例：

```text
疑似：双刀流
识别不确定，点击修正
```

## 23. 截图黑屏处理

显示器级截图需要保留黑屏检测。黑屏判断不能只看整张图是否全黑，还需要结合画面中心区域和用户校准区域采样，避免把加载画面、暗色背景或短暂切屏误判为失败。

如果助手检测到截图为黑屏：

```text
停止 OCR
延迟 100 - 300ms 后重新截图
提示用户切换无边框显示模式
提供手动输入兜底
```

提示文案：

```text
自动识别失败：当前截图为黑屏。
建议将 LOL 显示模式切换为无边框，或按快捷键手动输入海克斯。
```

## 24. 手动兜底模式

助手需要保留手动输入模式。

适用情况：

```text
截图黑屏
OCR 识别失败
用户未完成校准
特殊分辨率 / 显示模式不兼容
```

交互方式：

```text
按快捷键呼出手动输入
输入当前三个海克斯名称
助手展示对应说明
```

手动模式不影响自动模式，作为异常场景下的可用性保障。

## 25. 阶段划分

当前路线按阶段推进，避免把尚未验证的能力直接放进完整状态机。

已完成：

```text
显示器级截图验证
基础黑屏检测验证
延迟截图验证
中心区域采样验证
```

下一阶段优先做 POC：

```text
用户校准区域
OCR 识别与词库纠错
透明、置顶、点击穿透展示层
独占全屏 Overlay 可行性
本地 Live Client Data API 可用性检测
```

POC 通过后，再接入：

```text
ApexLOL 数据解析
本地缓存
完整海克斯选择状态机
多档位连续处理
重随变化处理
```

如果 Overlay POC 在独占全屏不稳定，只能回退到无边框覆盖层、独立助手窗口或手动兜底，不能转向注入、Hook 或读取内存。

## 27. 核心运行流程

完整运行流程如下：

```text
助手启动
  ↓
静默托盘运行
  ↓
检测本地接口是否可用
  ↓
读取本地接口返回的有效局内数据
  ↓
读取当前英雄和等级
  ↓
根据等级生成 pendingMilestoneQueue
  ↓
如果没有待处理海克斯：
  继续轻量监听
  ↓
如果存在待处理海克斯：
  低频显示器级截图并检测海克斯展示 / 隐藏按钮
  ↓
卡片已展开：
  直接进入展开判断，允许 OCR 和展示层
  ↓
卡片未展开且按钮不存在：
  继续等待
  ↓
卡片未展开且按钮存在：
  确认处于海克斯选择阶段
  ↓
判断卡片是否展开
  ↓
卡片未展开：
  隐藏说明层
  继续等待展开
  ↓
卡片已展开：
  OCR 识别三个名称区域
  词库纠错
  根据英雄 + 海克斯查询 ApexLOL
  在三张卡片下方展示说明层
  ↓
持续低频 OCR
  ↓
如果某个 slot 名称变化：
  判断为重随
  刷新该 slot 展示层
  ↓
如果当前轮选择完成：
  标记当前 milestone 已完成
  检查 pendingMilestoneQueue
  ↓
如果还有后续档位：
  继续等待下一轮海克斯
  ↓
如果没有后续档位：
  等待按钮消失
  ↓
按钮消失：
  隐藏说明层
  退出海克斯选择阶段
  回到普通监听
```

## 28. 核心规则汇总

```text
等级决定是否有资格
队列决定是否还有未处理档位
卡片决定是否展开和是否需要 OCR
卡片不可见时，按钮决定是面板收起还是阶段结束候选
名称变化决定是否重随
底部锚点决定展示层位置
卡片不可见后的按钮消失才可作为整个选择阶段结束候选
```

更具体地说：

```text
卡片消失，不代表选择完成
卡片可见优先于按钮区域，按钮被详情浮层遮挡不影响 OCR
按钮存在，说明选择阶段仍在
按钮存在但卡片不可见，说明面板被收起
卡片可见时即可进行 OCR
卡片不可见时按钮消失，才允许结束当前选择阶段
待处理队列可能包含多个档位
多档位场景下，需要连续处理多轮选择
```

## 29. 最终一句话总结

LOL 海克斯乱斗助手的核心设计是：

```text
用 Riot / LOL 本地只读接口识别英雄和等级；
用等级生成待处理海克斯队列；
用显示器级截图和用户校准区域检测选择阶段；
用用户校准的按钮区域判断是否仍处于海克斯选择阶段；
用用户校准的名称区域 OCR 识别三个海克斯；
用用户校准的底部锚点区域定位透明、置顶、点击穿透展示层；
用 ApexLOL 实时查询英雄 × 海克斯联动信息；
用 OCR 变化处理重随；
用待处理队列处理多档连续选择；
用按钮消失判断整个选择阶段结束。
```
