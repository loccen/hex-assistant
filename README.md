# LOL 海克斯乱斗助手 POC

这是 LOL 海克斯乱斗助手的当前 POC 版本，用来验证“显示器级截图 + 用户校准 + OCR + 本地接口状态机 + Overlay 展示”的技术路线。当前项目已经不只是第一阶段截图 Demo，但仍处在实测和联调阶段，还不能按完整可用产品交付。

项目目标是给玩家提供局内信息展示和人工决策辅助，不替玩家点击、不替玩家选择，也不读取游戏进程内部数据。

## 低可疑面边界

为了降低对游戏反作弊组件的干扰，当前实现继续保持低可疑面：

- 不扫描游戏进程。
- 不枚举窗口标题。
- 不按游戏名称查找窗口。
- 不注入、不挂钩、不读取游戏内存。
- 不模拟点击、不自动选择、不替玩家确认操作。
- 只使用显示器级截图、用户校准区域、OCR、本地允许接口和本地缓存。

## 当前已实现

- 截图诊断：支持目标显示器截图、主显示器截图、中心区域截图，导出日志、JSON 和 PNG 样本，用于判断截图是否成功、是否黑屏。
- 校准与区域保存：支持保存截图尺寸、归一化坐标、名称 OCR 区域、底部锚点区域、展示 / 隐藏按钮区域，并可重新加载。
- Overlay POC：支持按校准锚点渲染三张说明卡片，用于验证位置、显示和隐藏逻辑。
- OCR POC：支持对校准后的名称区域做 OCR，并结合词库纠错得到标准海克斯名称；运行时会保存 slot 原始裁剪图、名称行裁剪图和增强裁剪图，方便判断定位是否偏移。
- Live Client API 与状态机 POC：支持通过本地允许接口读取状态信息，并用状态机区分普通监听、等级满足、面板收起、面板展开、当前轮处理等状态。
- ApexLOL 后端查询 / 缓存接口：支持按英雄和海克斯查询后端接口并写入缓存，前端查询展示 POC 已实现；真实来源解析稳定性、Overlay 局内展示和真实样本验收仍待完成。

## 未完成和待实测

- 真实 Windows 桌面环境下的 Overlay 可见性和点击穿透仍需实测，尤其是游戏窗口、无边框和全屏场景。
- OCR 在真实游戏样本中的稳定性、低置信度处理和词库纠错效果仍需继续验证。
- ApexLOL 查询缓存和前端查询展示 POC 已实现，真实来源解析稳定性、Overlay 局内展示和真实样本验收仍待完成。
- 完整自动状态机还未完成，当前是 POC 级流程验证，不能视为最终自动化逻辑已经通过验收。
- 多显示器、分辨率变化、UI 缩放变化后的完整回归仍需补齐。

当前最终主分支仍只是 POC 级实现，最终验收需要按各场景人工复测。

ApexLOL 解析当前采用保守解析策略；请求失败、页面结构不匹配或解析不到可靠内容时，必须显示“暂无数据”，不能把这部分当作最终真实效果验收通过。

## 当前截图实测证据

以下只记录当前已观察到的测试证据，不代表所有机器、地图或后续版本的永久结论：

- 不同地图和显示模式需要分别验收，不能只用某一次成功截图推断全部场景通过。
- 截图内容哈希与历史成功报告完全重复时，存在旧帧风险，不能按“截图成功”直接判定通过。
- 2026-04-26 的当前实测证据来自
  `/mnt/c/Users/loccen/AppData/Roaming/dev.local.screen-capture-diagnostic/diagnostics`。
- 当前实测证据显示：莲华栈桥、嚎哭深渊、屠夫之桥三张地图的无边框模式均出现连续截图刷新，结果为 `black=false` 且 `stale=false`。
- 当前实测证据显示：三张地图的全屏模式都没有通过连续两次刷新标准。莲华栈桥全屏拿到过无边框旧帧；嚎哭深渊和屠夫之桥全屏首张非黑屏，但第二张命中旧帧。
- 当前 MVP 后续 OCR、Overlay 和状态机真实验收应优先限定在无边框模式；独占全屏只保留为风险和研究项，不作为默认可用前提。

## 运行环境

- 项目运行时由 `mise.toml` 固定：
  - Node.js `22.15.1`
- Rust `stable`
- WSL 可以用于开发和前端构建。
- 真实截图、Overlay 和点击穿透验证必须在 Windows 桌面环境运行，因为 WSL 进程不能直接捕获 Windows 桌面画面。

## 安装依赖

```bash
mise trust
mise install
mise exec -- npm install
```

## 开发运行

在 Windows 桌面环境进入项目目录后执行：

```bash
mise exec -- npm run tauri dev
```

## 构建验证

代码改动后使用：

```bash
mise exec -- npm run build
```

如果只修改文档，可以不运行构建，但仍应运行 `git diff --check` 并检查关键状态描述。

## 建议验证流程

### 截图诊断

请分别测试以下场景：

1. 客户端或桌面目标界面
2. 游戏内窗口模式
3. 游戏内无边框模式
4. 游戏内全屏模式

每个场景按下面步骤操作：

1. 把目标画面切到要测试的显示模式。
2. 打开助手。
3. 优先选择“主显示器”。
4. 如果是游戏内全屏模式，把“延迟截图”设为 8 秒或 12 秒。
5. 在诊断区填写显示模式备注和地图备注。
6. 点击“运行截图诊断”后立刻切回目标画面，等它自动截图。
7. 记录页面显示的结论，特别是黑屏判断、旧帧判断和历史匹配报告。
8. 打开页面中的导出目录，检查 `diagnostic.log`、`diagnostic.json` 和 PNG 样本。

### 校准、Overlay 和 OCR

1. 使用可见三张海克斯卡片的样本完成校准。
2. 保存并重新加载校准配置，确认框选区域仍落在正确位置。
3. 运行 Overlay POC，确认三张说明卡片位于对应海克斯卡片下方。
4. 在真实 Windows 桌面环境中测试点击穿透，不要只看开发窗口效果。
5. 对真实游戏样本运行 OCR，记录原始识别结果、置信度和词库纠错结果。
6. 悬停符文让详情浮层遮挡底部按钮区域时，确认三张卡片仍展开、Overlay 不隐藏、OCR 不停止。

已有无边框截图样本时，可以不重新进游戏，直接离线复盘 OCR 定位：

```bash
mise exec -- node scripts/replay-ocr-debug.mjs \
  /mnt/c/Users/loccen/AppData/Roaming/dev.local.screen-capture-diagnostic/calibration/profile.json \
  /mnt/c/Users/loccen/AppData/Roaming/dev.local.screen-capture-diagnostic/calibration/snapshots/calibration-20260426-182847-monitor-0.png
```

脚本会在 `ocr-debug-replay/` 下生成三槽 raw、focused、enhanced PNG 和 `replay-report.json`；如果离线 OCR 环境不可用，也会保留裁剪坐标和图片路径，便于人工查看定位。

## 反馈内容

每个场景至少反馈：

- 显示模式：窗口 / 无边框 / 全屏
- 地图：嚎哭深渊 / 莲华栈桥 / 屠夫之桥，或其他实际地图
- 页面结论
- 哪个截图策略成功
- 是否疑似黑屏
- 是否疑似旧帧，以及匹配到的历史报告
- 校准配置是否能保存和恢复
- Overlay 是否可见，是否影响点击
- OCR 原始结果、置信度和词库纠错结果
- `diagnostic.log`
- 对应 PNG 样本

日志里会包含系统版本、Rust target、应用数据目录、显示器信息、显示模式备注、地图备注、截图尺寸、耗时、亮度统计、黑屏判断、旧帧判断、历史匹配报告、样本路径和错误信息。

## 当前诊断策略

- `xcap_requested_target`：按页面选择的目标截图。
- `xcap_primary_monitor`：始终截主显示器，用于验证全屏画面。
- `xcap_center_region`：截主显示器中心区域，用于快速判断画面是否真实可见或全黑。

如果全屏模式下三个策略都黑屏，后续流程应提示切换无边框，并保留手动输入兜底。
