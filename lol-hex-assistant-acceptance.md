# LOL 海克斯乱斗助手验收文档

## 1. 文档定位

本文档是 LOL 海克斯乱斗助手后续实现、测试和交付的最终验收主文档。阶段任务是否完成，以本文档中的验收项、禁止事项、人工可复现场景、日志和样本要求为主。

`lol-hex-assistant-design.md` 负责解释产品逻辑、技术约束和核心规则。本文档不重复展开设计理由，只把设计要求转成可以人工复现、可以检查结果、可以判断通过或不通过的验收标准。

任何阶段如果出现本文档没有覆盖但会影响用户安全、游戏公平性、反作弊风险或局内可用性的情况，按更严格的规则验收。

## 当前实现状态

截至当前文档更新，主分支已完成 Stage 2A-D 的 POC 实现：截图诊断、校准截图和区域保存、Overlay POC、OCR POC、Live Client API 与状态机 POC；Stage 2E 已实现 ApexLOL 查询缓存和前端展示 POC。这里的“已完成”表示已有可运行的 POC 功能和基础流程，不等于所有真实游戏场景已经验收通过。

当前仍需后续完成或实测：

- 真实 Windows 桌面环境下的 Overlay 可见性和点击穿透，尤其是窗口、无边框、全屏三类显示模式。
- OCR 引擎已定为 RapidOCR + ONNXRuntime（`scripts/ocr_sidecar.py`），离线回归评估 15/15 通过。Tesseract 只保留为调试基线。仍需在真实游戏无边框样本上验收，覆盖不同地图、缩放、画质。
- ApexLOL 真实来源解析稳定性、Overlay 联动和真实样本验收。
- 完整自动状态机，包括多档位连续处理、重随后的 slot 级刷新、异常恢复和长时间局内运行。

文档或代码改动后的常用命令仍以 `mise` 为准：

```bash
mise exec -- npm run build
mise exec -- npm run tauri dev
```

只修改文档时不要求运行 npm 构建，但必须至少通过 `git diff --check`，并建议用 `rg` 检查关键状态描述是否准确。

## 2. 总体验收边界

### 2.1 必须满足

- 助手只提供信息展示和人工决策辅助，不替玩家做选择。
- 所有自动识别都必须来自允许的本地数据接口、显示器级截图、用户校准区域、OCR 和本地缓存。
- 所有展示内容必须能被用户理解，必须说明数据来源是 ApexLOL。
- 每个阶段都必须留下可复查的日志、配置或样本，不能只靠开发者口头说明。
- 涉及截图、OCR、Overlay、状态机和数据查询的结论，都必须能在人工测试流程中复现。

### 2.2 禁止事项

- 禁止新增注入游戏进程的实现。
- 禁止新增 Hook、DLL 注入、驱动级读取、内存读取或修改。
- 禁止读取 LOL 游戏内存、扫描关键内存地址或解析进程内部对象。
- 禁止自动点击、自动选择、自动重随、自动确认或模拟玩家输入。
- 禁止绕过游戏客户端、反作弊组件或系统权限限制。
- 禁止为了识别海克斯而持续高频全屏截图或持续高频 OCR。
- 禁止在没有用户校准或明确兜底输入的情况下伪造识别结果。

### 2.3 通用通过标准

- 测试人员可以按文档步骤独立复现，不依赖开发者本机环境。
- 同一个场景至少连续复测 3 次，结果一致或差异有明确日志说明。
- 异常场景必须有可理解的提示，例如截图黑屏、OCR 低置信度、ApexLOL 请求失败。
- 失败时不能影响玩家操作，不能遮挡关键按钮，不能拦截游戏点击。

## 3. Stage 2A 截图与校准基础

目标：确认助手能在用户授权和人工操作下获取可用截图样本，并完成后续 OCR、按钮检测和 Overlay 定位所需的校准配置。

### 3.1 必须满足

- 能分别记录窗口、无边框、全屏三类显示模式的截图诊断结果。
- 能记录本次诊断填写的显示模式备注和地图备注。
- 截图样本必须包含目标画面尺寸、亮度统计、黑屏判断、截图耗时和保存路径。
- 截图样本必须记录旧帧判断；成功截图如果与历史成功截图哈希完全相同，必须提示疑似旧帧和匹配到的历史报告。
- 全屏截图样本不得只保存全黑画面；如果全屏不可用，必须明确提示建议切换无边框。
- 疑似旧帧的全屏截图不得只按“非黑屏”判定通过，必须结合 PNG 样本、地图备注、显示模式备注和历史匹配记录人工确认。
- 校准向导必须支持框选三类区域：
  - 左、中、右三张海克斯名称 OCR 区域。
  - 左、中、右三张海克斯卡片底部锚点区域。
  - 底部海克斯展示 / 隐藏按钮区域。
- 校准配置必须保存原始截图尺寸和归一化坐标，重新加载后能还原真实坐标。
- 校准完成后必须能在截图样本上回显框选区域，便于人工检查是否框到正确位置。
- 用户修改分辨率、显示模式、UI 缩放或显示器后，必须提供重新校准入口。

### 3.2 禁止事项

- 禁止通过枚举游戏窗口、扫描游戏进程或读取进程内存来代替显示器级截图。
- 禁止在全屏截图失败时继续执行 OCR 并给出看似正常的结果。
- 禁止只存绝对坐标，不存归一化坐标。
- 禁止把校准区域硬编码为单一分辨率。
- 禁止在未完成校准时进入自动 Overlay 展示流程。

### 3.3 人工可复现场景

- 场景 A：窗口模式下运行截图诊断，导出的 PNG 能看到目标画面，不是黑屏，日志中的黑屏判断为否。
- 场景 B：无边框模式下运行截图诊断，PNG 能看到海克斯选择界面或桌面目标画面，截图尺寸与显示器尺寸一致。
- 场景 C：全屏模式下设置 8 秒或 12 秒延迟截图，点击诊断后切回游戏；如果截图成功，样本非黑屏且没有匹配历史旧帧；如果失败，页面和日志都提示全屏不可用并建议无边框。
- 场景 D：在一张 1920x1080 样本上框选 3 个名称区域、3 个底部锚点和 1 个按钮区域，保存后关闭应用再打开，区域位置保持一致。
- 场景 E：将同一套归一化坐标应用到相同比例的另一张样本，区域仍落在对应卡片和按钮附近。
- 场景 F：同一显示模式至少覆盖嚎哭深渊、莲华栈桥和屠夫之桥等实际地图备注；如果某个地图出现黑屏或旧帧风险，必须单独保留该地图的诊断报告。

### 3.4 日志 / 样本要求

- 每次截图诊断至少导出 `diagnostic.log`、`diagnostic.json` 和 PNG 样本。
- 日志必须包含显示器信息、显示模式备注、地图备注、截图策略、截图尺寸、耗时、亮度统计、黑屏判断、旧帧判断、历史匹配报告、错误信息和样本路径。
- 校准结果必须导出为可读 JSON，包含版本号、截图尺寸、语言、三个名称区域、三个底部锚点、按钮区域和 Overlay 间距配置。
- 校准回显样本必须能看出每个区域对应的 slot 编号。

## 4. Stage 2B Overlay POC

目标：确认 Overlay 能显示在三张海克斯卡片正下方，跟随校准锚点定位，并且不影响玩家操作。

### 4.1 必须满足

- Overlay 必须分别渲染左、中、右三个说明卡片，位置来自对应底部锚点区域。
- 每个说明卡片必须显示评级、英雄 × 海克斯名称、一句话说明、一句话提醒和数据来源。
- Overlay 必须能在卡片展开时显示，在面板收起或选择阶段结束时隐藏。
- Overlay 不得拦截游戏点击；用户点击海克斯卡片、重随按钮、展示 / 隐藏按钮时，游戏必须正常响应。
- Overlay 不得遮挡海克斯名称、重随按钮、选择按钮或底部展示 / 隐藏按钮。
- 多显示器环境下必须明确 Overlay 绑定的目标显示器，并在日志中记录。
- 当校准缺失或坐标异常时，必须隐藏 Overlay 并提示重新校准。

### 4.2 禁止事项

- 禁止通过注入、Hook 或修改游戏窗口来绘制 Overlay。
- 禁止创建会抢占输入焦点、拦截鼠标或阻止游戏热键的窗口。
- 禁止让 Overlay 默认覆盖整张卡片或挡住玩家选择入口。
- 禁止在无法确认卡片展开时强行展示说明卡片。
- 禁止在 Overlay 异常时自动点击或自动修正游戏状态。

### 4.3 人工可复现场景

- 场景 A：使用一张完成校准的样本，Overlay 三张说明卡片分别出现在三张海克斯卡片正下方。
- 场景 B：打开游戏海克斯面板，鼠标点击三张卡片和底部展示 / 隐藏按钮，Overlay 不拦截点击，游戏响应正常。
- 场景 C：点击底部按钮收起海克斯面板，Overlay 在配置的隐藏时间内消失；再次展开后重新显示。
- 场景 D：调整 Overlay 间距配置，三张说明卡片 y 位置同步变化，但 x 和宽度仍跟随对应底部锚点。
- 场景 E：删除或破坏校准配置后启动助手，Overlay 不显示，并提示需要重新校准。

### 4.4 日志 / 样本要求

- 日志必须记录 Overlay 创建时间、目标显示器、窗口点击穿透状态、三张说明卡片的 x / y / width / height。
- 至少保留一张 Overlay 展示样本和一张面板收起后 Overlay 隐藏样本。
- Overlay 点击穿透测试必须记录测试时间、测试动作和结果，例如“点击 slot2 卡片，游戏收到点击，Overlay 未获得焦点”。
- 异常日志必须能区分校准缺失、坐标越界、目标显示器不存在和 Overlay 创建失败。

## 5. Stage 2C OCR 与词库纠错

目标：确认助手只在需要时处理三个名称区域，并具备可复盘的裁剪定位、样本回放、调试图和候选输出能力。Tesseract 当前只作为 POC / 调试基线使用，不作为最终 OCR 核心方案；OCR 引擎需要后续单独评估或替换。

### 5.1 必须满足

- OCR 只对用户校准的三个名称区域执行，不对整屏持续识别。
- 三张卡片展开时，允许 OCR 三个名称区域；按钮区域被符文详情浮层遮挡不能阻止 OCR。
- 必须保存 raw、focused、tight-line、enhanced 等裁剪或增强候选图，并输出每个候选的 OCR 文本、置信度、匹配分和最终选择。
- OCR 原始结果必须经过保守的词库匹配；只有置信度和相似度达标时才能输出标准海克斯名称，不能直接拿原始字符串查询 ApexLOL。
- 低置信度时必须显示疑似结果，并允许用户手动修正。
- 低置信度或未匹配时必须保留定位调试图和候选输出，用于判断是定位问题、预处理问题还是 OCR 引擎问题。
- 某个 slot 识别失败时，必须保留旧内容；连续多次失败后进入异常提示。
- 重随后如果只有一个 slot 名称变化，只能刷新对应 slot 的识别结果和展示内容。
- OCR 频率必须是低频，例如 800 - 1200ms 一次，不能变成持续高频识别。
- 后续正式验收必须用多张真实样本覆盖不同地图、画质、缩放和 UI 状态；如果 Tesseract 继续低稳，必须切换本地 OCR 引擎适配层。

### 5.2 禁止事项

- 禁止在普通监听状态下持续 OCR。
- 禁止对全屏截图执行无差别 OCR 后再搜索名称。
- 禁止 OCR 失败时编造海克斯名称。
- 禁止低置信度结果直接覆盖高置信度旧结果。
- 禁止把用户手动修正写死为全局规则，影响其他海克斯名称。
- 禁止通过针对单个样本的 OCR 错字映射、符文名特化混淆字表或低置信度强行匹配来制造通过结果。
- 禁止把某一张样本的 Tesseract 成功识别当作 Stage 2C 最终通过结论。

### 5.3 人工可复现场景

- 场景 A：优先使用无边框模式样本，离线回放输出三个 slot 的裁剪图路径、候选文本、置信度、匹配分和最终选择。
- 场景 B：准备多张真实样本，覆盖不同地图、画质、缩放和 UI 状态，记录 Tesseract 或替代引擎的稳定性。
- 场景 C：准备一张低清或遮挡样本，低置信度时界面显示“疑似结果”并允许手动修正。
- 场景 D：先识别三张名称，再把第二张名称换成另一张海克斯样本；助手只刷新 slot2，slot1 和 slot3 内容保持不变。
- 场景 E：收起海克斯面板后，按钮仍存在但卡片不可见；助手停止 OCR，隐藏说明层，不标记当前轮完成。
- 场景 F：鼠标悬停某张符文导致详情浮层遮挡底部按钮区域，但三张卡片仍展开；助手继续 OCR，不隐藏 Overlay，不标记当前轮完成。

### 5.4 日志 / 样本要求

- 每次 OCR 必须记录触发原因、slot 编号、裁剪区域坐标、候选图路径、原始结果、置信度、词库匹配结果和耗时。
- 词库纠错日志必须记录候选名称和最终选择，便于人工判断为什么修正。
- 已有 profile.json 和截图 PNG 时，必须能用离线复盘命令生成三槽 raw、focused、tight-line、enhanced 等调试图；离线 OCR 不可用时也必须输出坐标和文件路径。
- 重随日志必须记录变化前后的 slot 名称，例如 `slot2: 吞噬灵魂 -> 巨像勇气`。
- 至少保留三类 OCR 样本：正常识别、词库纠错、低置信度手动修正。

## 6. Stage 2D 本地接口与状态机

目标：确认助手能通过本地接口获取游戏状态、英雄和等级，并按状态机正确处理海克斯选择阶段、多档位、面板收起和重随。

### 6.1 必须满足

- 本地接口只用于获取允许读取的游戏状态、当前英雄和当前等级。
- 状态机必须包含普通监听、等级满足、选择阶段活跃、面板收起、面板展开、当前轮完成、阶段完成等状态。
- 等级达到 3 / 7 / 11 / 15 时，必须根据已完成档位生成待处理队列。
- 卡片 / 名称区域可见优先于按钮区域；按钮区域只在卡片不可见时用于区分面板收起、入口还在或选择阶段结束候选。
- 按钮存在但卡片不可见时，必须进入面板收起状态，隐藏 Overlay，不标记当前轮完成。
- 卡片可见时，必须进入面板展开状态，允许 OCR 和展示；按钮区域被符文详情浮层遮挡不能改变展开状态。
- 卡片消失不等于整个选择阶段结束；只有卡片不可见时，按钮消失才允许作为结束当前选择阶段的候选依据。
- 多档位场景必须能连续处理多轮选择，例如当前等级 11 且 7、11 两档未完成时，需要依次处理两轮。
- 重随后只刷新发生变化的 slot，不得重置整个阶段或错误标记档位完成。

### 6.2 禁止事项

- 禁止把卡片收起当成选择完成。
- 禁止只维护单个待处理档位，导致多档位场景丢失。
- 禁止按钮从未出现时就把阶段标记为完成。
- 禁止本地接口读取游戏内存或使用未授权进程能力。
- 禁止状态异常时自动点击、自动展开面板或自动选择海克斯。

### 6.3 人工可复现场景

- 场景 A：玩家等级低于 3 级时，助手只处于普通监听状态，不截图、不 OCR、不展示 Overlay。
- 场景 B：玩家达到 3 级且存在未完成档位，助手进入等级满足状态，只低频检测按钮区域。
- 场景 C：按钮出现但卡片未展开，助手进入面板收起状态，Overlay 隐藏，不标记完成。
- 场景 D：三张卡片展开，助手进入面板展开状态，OCR 三张名称并展示说明；按钮区域被详情浮层遮挡不能阻止该状态。
- 场景 E：点击底部按钮收起再展开，状态在面板收起和面板展开之间切换，当前轮不被误判完成。
- 场景 F：当前等级 11，已完成 3 级，待处理队列为 7、11；第一轮选择完成后移除 7，继续等待第二轮，第二轮完成后再等待按钮消失。
- 场景 G：slot2 重随后名称变化，状态机保持在当前选择阶段，只刷新 slot2，不刷新 slot1 和 slot3。
- 场景 H：鼠标悬停符文导致详情浮层遮挡底部按钮区域，但三张卡片仍展开；助手保持面板展开，不隐藏 Overlay，不停止 OCR，不标记当前轮完成。

### 6.4 日志 / 样本要求

- 状态机日志必须记录每次状态迁移的来源状态、目标状态、触发条件和时间。
- 本地接口日志必须记录英雄、等级、接口返回时间和错误信息，但不能记录敏感令牌。
- 待处理队列日志必须记录等级、已完成档位、队列生成结果和队列变更。
- 多档位测试必须保留完整日志，能看出每一轮如何开始、如何完成、何时等待按钮消失。
- 重随测试必须保留 slot 级变化日志，证明只刷新对应 slot。

## 7. Stage 2E ApexLOL 查询与展示

目标：确认助手能用英雄和标准海克斯名称查询 ApexLOL，解析需要的信息，并在 Overlay 中用短内容展示。

### 7.1 必须满足

- 查询条件必须来自当前英雄和词库纠错后的标准海克斯名称。
- 查询结果必须按“英雄 + 海克斯”粒度缓存。
- 命中缓存时必须先展示缓存内容，并在日志中标记缓存命中。
- 未命中缓存时请求 ApexLOL，解析评级、机制说明、提醒、来源页面地址和更新时间。
- 请求失败、页面结构变化或没有数据时，必须展示“暂无数据”或同等明确提示，不能空白或误导。
- Overlay 中必须显示“数据来源：ApexLOL”，并提供完整来源页面入口。
- 展示文案必须短，适合局内快速扫读；不能用大段解释遮挡游戏画面。
- 评级必须使用设计文档中的五档文案：夯爆了、顶级、人上人、NPC、拉完了。

### 7.2 禁止事项

- 禁止完整爬取 ApexLOL 源站并维护全量数据。
- 禁止把没有来源的数据伪装成 ApexLOL 结果。
- 禁止请求失败时沿用其他英雄或其他海克斯的结果。
- 禁止缓存粒度只按海克斯名称，忽略英雄差异。
- 禁止展示层把来源入口做成会拦截游戏点击的控件。
- 禁止自动替玩家选择评级最高的海克斯。

### 7.3 人工可复现场景

- 场景 A：英雄为蔚，OCR 得到三个标准海克斯名称；助手分别查询并展示三张说明卡片。
- 场景 B：重复查询同一个英雄 + 海克斯组合，第二次命中缓存，展示速度更快，日志标记缓存命中。
- 场景 C：断网或 ApexLOL 请求失败，Overlay 对对应 slot 显示暂无数据，不影响其他已成功 slot。
- 场景 D：ApexLOL 页面无法解析时，日志记录解析失败原因，界面提示暂无数据，并保留来源入口或错误说明。
- 场景 E：slot2 重随为新海克斯后，只查询 slot2 的英雄 + 新海克斯组合，slot1 和 slot3 不重新请求。

### 7.4 日志 / 样本要求

- 查询日志必须记录英雄、海克斯名称、请求 URL、缓存命中状态、请求耗时、解析状态和错误信息。
- 缓存记录必须包含英雄、海克斯名称、评级、标签或说明、来源、来源 URL 和获取时间。
- 展示样本必须至少覆盖成功结果、缓存命中、请求失败、无数据四类。
- 任何网络错误都必须有可读日志，不能只在控制台抛异常。

## 8. 覆盖测试计划

### 8.1 截图测试

- 覆盖窗口、无边框、全屏三类显示模式。
- 覆盖主显示器和至少一个非主显示器。
- 覆盖截图成功、截图黑屏、截图策略失败三类结果。
- 检查导出的 PNG 样本、JSON 和日志是否能互相对应。

### 8.2 Overlay 测试

- 覆盖三张说明卡片的定位、显示、隐藏、重新显示。
- 覆盖点击穿透，确认 Overlay 不拦截卡片、按钮和游戏热键。
- 覆盖校准缺失、坐标越界、多显示器目标错误。
- 覆盖面板收起后隐藏和再次展开后恢复。

### 8.3 校准测试

- 覆盖首次校准、保存、重新加载、重新校准。
- 覆盖归一化坐标还原。
- 覆盖名称区域、底部锚点、按钮区域的人工回显检查。
- 覆盖分辨率或 UI 缩放变化后的失效提示。

### 8.4 OCR 测试

- 覆盖三张名称正常识别。
- 覆盖词库纠错、低置信度、手动修正。
- 覆盖离线样本回放、调试图路径和候选输出。
- 覆盖多张真实样本的 OCR 引擎评估；Tesseract 低稳时记录替换本地 OCR 引擎适配层的结论。
- 覆盖单个 slot 识别失败时保留旧内容。
- 覆盖重随后只刷新对应 slot。
- 覆盖按钮存在但卡片收起时停止 OCR。
- 覆盖符文详情浮层遮挡按钮区域但卡片仍展开时继续 OCR。

### 8.5 状态机测试

- 覆盖等级低于 3、达到 3 / 7 / 11 / 15、多档位积累。
- 覆盖按钮不存在、按钮存在、卡片展开、卡片收起、按钮消失，以及按钮区域被详情浮层遮挡但卡片仍展开。
- 覆盖当前轮完成和整个选择阶段完成的区别。
- 覆盖待处理队列生成、移除和清空。
- 覆盖异常恢复，例如 OCR 连续失败、截图黑屏、接口暂时不可用。

### 8.6 数据测试

- 覆盖 ApexLOL 查询成功、缓存命中、请求失败、解析失败、没有数据。
- 覆盖不同英雄使用同一个海克斯时缓存不串数据。
- 覆盖同一英雄不同海克斯时展示内容独立。
- 覆盖来源 URL 和“数据来源：ApexLOL”的展示。

### 8.7 安全边界测试

- 检查代码和依赖中没有新增注入、Hook、内存读取、自动点击、自动选择相关实现。
- 检查运行日志中没有进程内存读取、窗口注入、模拟点击或反作弊绕过记录。
- 检查用户交互流程中没有任何自动替玩家选择海克斯的入口。
- 检查异常兜底只提供提示、手动输入或手动修正，不执行游戏操作。
